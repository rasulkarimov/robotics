# План развития робота (car + xArm)

Составлен 2026-07-11 по итогам сессии, в которой заработали: управление шасси,
камера, xArm по USB HID, и захват предмета обучением по демонстрации (`replay grab`
/ `replay place`, 4 успеха из 5 циклов).

## Текущее состояние

| Компонент | Статус |
|---|---|
| Шасси (моторы, руль) | работает, `car.py` |
| Камера (mjpg-streamer, порт 8090) | работает, но качество низкое; периодически виснет |
| xArm (6 сервоприводов, USB HID) | работает, `arm.py`, карта суставов подтверждена |
| Захват предмета | работает через обучение по демонстрации |
| Ultrasonic | датчик читается напрямую (`mDev.py ultrasonic`), но через сервер вешает его |
| Связка рука+шасси | **нет** — работают независимо |

## Известные проблемы, которые блокируют развитие

1. **Нет восприятия глубины.** Одна 2D-камера → навести клешню на предмет «на глаз»
   невозможно (4 промаха подряд). Обошли обучением по демонстрации, но это работает
   только для предмета в заранее показанной точке.
2. **Укладка неточная.** Предмет постепенно уползает от исходной точки, и через
   несколько циклов очередной захват промахивается.
3. **Сервер однопоточный.** Запрос `>Ultrasonic` при плохой проводке датчика блокирует
   весь сервер — теряется и управление, и камера.
4. **Pi спонтанно перезагружается** (много раз за сессию), сервер не поднимается сам.

## 1. Лучшая камера

**Зачем:** текущая даёт размытую картинку вблизи (не фокусируется), и это главная
причина, по которой захват «по зрению» не работает.

**Что сделать:**
- Поставить камеру с нормальным ближним фокусом (или макро). Мёртвая зона фокуса —
  это то, из-за чего клешня и предмет всегда не в фокусе одновременно.
- **Позиция важнее модели.** Камера на самой клешне бесполезна для захвата (предмет
  уходит из кадра в момент смыкания). Нужен один из вариантов:
  - камера на «плече»/локте, смотрящая на рабочую зону сверху-сбоку → видно и клешню,
    и предмет одновременно;
  - либо стационарная камера сбоку от рабочей зоны.
- Разрешение поднять: сейчас 480x320. Для распознавания предметов нужно хотя бы 640x480.

**Что это даст:** возможность считать координаты предмета в кадре и наводиться по
обратной связи, а не воспроизводить заученную траекторию.

## 2. Ultrasonic

**Зачем:** дешёвая и надёжная оценка расстояния — ровно то, чего не хватает камере.

**Что сделать:**
- Разобраться с проводкой (сейчас через сервер запрос вешает всё намертво; напрямую
  через `mDev.py ultrasonic` датчик читается нормально — значит дело в сервере/таймауте).
- **Обойти сервер:** читать датчик напрямую по I2C (`mDev.readReg(CMD_SONIC)`), не через
  TCP-команду `>Ultrasonic`. Тогда зависание сервера перестанет быть риском.
- Добавить в `car.py` чтение с таймаутом и защитой.

**Что это даст:**
- Безопасное движение шасси: проверять дистанцию перед каждым шагом вперёд
  (сейчас я въезжаю в мебель, потому что камера вблизи слепнет).
- Грубая оценка высоты/расстояния до предмета для руки.

## 3. Улучшение работы с xArm

**Приоритет — сделать захват устойчивым, а не только воспроизводимым.**

- **Библиотека поз.** Сейчас есть `grab`/`place` для одной точки. Нужно: обучить
  несколько точек (сетка на рабочей зоне), чтобы уметь брать предмет из разных мест.
- **Наведение по камере поверх обучения.** Гибрид: высоту и траекторию спуска берём из
  обученного движения (это работает), а горизонтальное положение (основание, серво 6)
  докручиваем по картинке. Это снимет главную слабость чистого воспроизведения.
- **Проверка захвата.** Не по числу с сервопривода (проверено: успех давал 526-597,
  промах 628 — границы перекрываются, ненадёжно), а по камере: предмет в клешне / исчез
  с пола.
- **Плавность.** Сейчас каждый ключевой кадр — отдельная команда с ожиданием.
  Промежуточная интерполяция сделает движение мягче и снизит риск сбить предмет.
- **Защита от упоров.** Клешня (серво 1) при команде «за предмет» останавливается и
  греется. Добавить проверку: если позиция не меняется — снять усилие.

## 4. Интеграция xArm с шасси

**Цель:** подъехать к предмету → взять → отвезти → положить. Сейчас это две
несвязанные системы.

**Порядок работ:**

1. **Общий инструмент.** Свести `car.py` и `arm.py` под один вызов (или общий модуль),
   чтобы последовательность «ехать → остановиться → взять» была одной командой,
   а не ручной склейкой.
2. **Питание.** Проверить, что рука и моторы не сажают шину одновременно. Pi уже
   перезагружался за сессию много раз — при одновременной нагрузке риск вырастет.
   Возможно, нужен отдельный источник для руки.
3. **Позиционирование шасси.** Ключевая проблема: рука обучена брать предмет в точке
   относительно **основания руки**. Значит шасси должно вставать в повторяемую позицию
   относительно предмета. Варианты:
   - подъезжать по ultrasonic до фиксированной дистанции (просто, надёжно);
   - доворачивать по камере, пока предмет не окажется в заданной точке кадра.
4. **Сценарий целиком:** найти предмет (камера) → подъехать (шасси + ultrasonic) →
   выровняться → `replay grab` → отъехать → `replay place`.

## Порядок (что раньше)

1. **Ultrasonic** — самое дешёвое и снимает главный риск (въезд в препятствия).
   Делается без нового железа.
2. **Камера** — разблокирует наведение по зрению; без неё всё остальное упирается
   в заученные траектории.
3. **Гибридное наведение руки** (обучение + камера) — прямое следствие пунктов 1-2.
4. **Связка с шасси** — последним, потому что требует всего вышеперечисленного
   (иначе шасси не сможет вставать повторяемо).
